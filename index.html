AWS Cloud Practitioner

AWS Cloud Practitioner

Tras haber obtenido esta certificación, tengo el conocimiento necesario para gestionar una infraestructura desplegada en esta nube. Si bien es la certificación más fundamental que ofrece AWS, te da una perspectiva bastante grande de lo que se puede hacer en esta plataforma.

Uno de mis proyectos en AWS ha sido levantar un Wordpress con alta disponibilidad y tolerante a fallos. Para ello, en primer lugar tenemos que entender como funciona Wordpress. Wordpress, resumiendo entre muchas comillas, es código PHP servido a través de un servidor web como Apache o Nginx, con una base de datos MySQL/MaríaDb por detrás. Entendiendo este funcionamiento, podemos usar los siguientes servicios:

  • Base de datos: Usaré el servicio RDS (Relational Database Service) de Amazon con MySQL para tener una base de datos en formato PaaS, es decir, solamente tengo que gestionar el motor de la base de datos en sí y los datos, del resto se encarga AWS simplificando la gestión.

  • Máquinas virtuales: necesitamos varias máquinas virtuales donde instalar nginx y php. Aquí se pondrán los ficheros de Wordpress y se conectarán a la RDS.

  • Balanceadores de carga y auto escalado: decidí usar dos zonas de disponibilidad para desplegar dos máquinas independientes. Utilizo dos zonas de disponibilidad para evitar caídas de servicio por fallo en los datacenters. Al utilizar dos máquinas es necesario poner un balanceador de carga delante para que distribuya el tráfico entre esas máquinas. También configuré un grupo de autoescalado para que, si en un momento hubiera más tráfico de lo normal, pudiera levantar una tercera máquina más como refuerzo y que cuando disminuyera la carga esta se destruyera automáticamente.

  • Sistema de ficheros compartidos: este servicio ofrece una unidad NFS que se puede montar tanto en máquinas EC2 como en el resto. Aquí es donde guardé tanto la configuración de Nginx como de wordpress y la monté en ambas máquinas. De este modo, da igual cuál se visitara y cuál se editará, ya que todas las máquinas cogen la misma unidad.

Migración Wordpress a Docker

Migración Wordpress a Docker

Uno de mis trabajos ha sido la migración de wordpress desde un proveedor ajeno a una infraestructura con máquinas virtuales en VMWare Esxi. Soy consciente de que existen varios plugins que hacen esta función, pero a mi, en lo personal, me gusta más realizarlo directamente, ya que así no se desconfigura nada y no dependo de una aplicación de terceros para poder restaurar.

En primer lugar levanté todas las máquinas virtuales que me hacían falta, en este caso una para la base de datos y otra para el wordpress en sí. La de wordpress tenía dos tarjetas de red, una con la que salía a Internet y con una IP fija, y la otra para comunicarnos mediante red privada con la base de datos. De esta manera pasamos de un punto de fallo (en un principio el wordpress estaba en una sóla máquina junto con la BBDD) a dos puntos de fallo, permitiendo más tolerancia a fallos.

Una vez tuve todas las máquinas creadas instalé Docker en cada una de ellas y levanté los contenedores que me hacían falta. Configuré todo y, por último, nos quedan los registros DNS. Para ello era simplemente cambié el registro de tipo A del dominio de la IP antigua a la nueva. En este punto tenía las dos infraestructuras funcionando, por lo que no hubo corte de servicio para los clientes en ningún momento.

Tras unos días monitorizando y comprobando que la infraestructura antigua ya no respondía peticiones, se dió de baja y se quedó funcionando la actual desplegada en contendores.

Creación de un portfolio desatendido y gratis

Creación de un portfolio desatendido y gratis

Otro proyecto que tenía en mente era crearme un portfolio. Me lo tomé como un entrenamiento con un doble propósito. En primer lugar, es muy recomendable tener un portfolio donde mostrar tus habilidades y proyectos, y por otro lado, me lo quisé tomar como un ejercicio para volver a repasar un poco Bootstrap, CSS y HTML, ya que lo tenía un poco olvidado.

El portfolio tenía que cumplir ciertas condiciones:

  • Tenía que estar hecho con un framework CSS, en mi caso elegí Bootstrap por la simpleza de uso y las funcionalidades que incluye.
  • Tenía que ser desatendido, es decir, tenía que estar alojado en un sitio que fuera gratis y que no tuviera URL excesivamente complicadas de recordar. Elegí GitHub Pages porque cumple con todas estas condiciones.
  • Al ser una página web estática, tenía que buscar la forma de introducir un formulario de contacto funcional. Para ello utilicé FormSpree, que, con su plan gratuito, permite 50 peticiones al formulario.

Al subirlo a GitHub, no me tengo que preocupar de certificados, ya que GitHub se podría definir como un PaaS y nosotros nos preocupamos solamente de subir contenido, ya que del resto se encarga GitHub. Por otra parte, tampoco tenemos que renovar dominios ni comprar nada, ya que usamos un subdominio de GitHub.
